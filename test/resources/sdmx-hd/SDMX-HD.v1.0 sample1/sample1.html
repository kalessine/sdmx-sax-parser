Sample 1 - How to exchange indicator definitions

What files are located in this folder?

The sample1 folder contains the following files and folders:

Generic SDMX Scheme Validation Files
  • These are the standard SDMX v2.0 scheme validation files that are use to perform generic validation of the XML files used in SDMX-HD. For easy of use and to avoid file path mismatches these files are duplicated into each folder that requires them.
Sample files
  • sample1.html - This web page
  • IMR Indicators 65,67,75 - Screenshots of three sample indicator definitions obtained from the WHO Indicator Metadata Registry. These definitions are used in the sample XML files.
  • GMS_INDICATOR.xml - The Generic MetaDataSet file contains metadata that needs to be exchanged for an Indicator
  • GMS_ORGANISATION.xml - The Generic MetaDataSet file contains metadata that needs to be exchanged for an Organisation
  • DSD.xml - The Data Structure Definition file holds structural information about the data being exchanged
  • MSD.xml - The MetaData Structure Definition file holds reference metadata about the data being exchanged

Why do we include code lists that are not needed?

One of the objectives of SDMX-HD is to reduce where possible the number or changes needed by each implementing system. The DSD files reference code lists used specifically for the exchange of data e.g. CL_GEOGRAPHIC_PLACE_NAME. To allow implementing systems to create one DSD file that can be used for both purposes (exchange of indicator defintions only or exchange of defintions and data), the code lists need to be retained to ensure the DSD files remain valid. Thus, whenever it does not make sense to include a code list (e.g. geographic place names are of no use to exchange indicator definitions), it is recommended to include the empty codelist file as placeholder. This also avoids the exchange of large amounts of data that are not actually used.

What does this sample showcase?

The WHO Indicator Metadata Registry (IMR) application is a repository of indicator definitions. Three indicators have been selected from this system and used in this sample.

  • An indicator of type count - Life expectancy at birth.
  • An indicator of type ratio - Neonatal mortality rate (per 1000 live births), with associated Numerator and Denominators
    • Numerator - Number of deaths during the first 28 completed days of life
    • Denominator - 1000 live births in a given year
  • An indicator of type percentage - Adults aged >= 15 years who are obese that does not have numerator or denominator specified. It is disaggregated as follows:
    • By Gender (from COMMON)
    • By Age Group (from CUSTOM/WHO)
    • By Location (from COMMON)
  • Disclaimer: Indicator definitions included in this sample should not be considered official or complete definitions from WHO. They are partial and may have been modified to illustrate some specific SDMX-HD concepts shown in this sample.

The Metadata for the sample indicators is:

  • Data type
  • Code
  • Description
  • Rationale
  • Method of Collection
  • Unit of Measure
  • Unit Multiplier
  • Topic
  • Sort Order
  • Version

How are these indicator definitions implemented in SDMX-HD?

The three sample indicators are implemented in the DSD,MSD and GMS files. The details of how this is achieved are listed below:

DSD File - This file contains the codelists, concepts, hierarchies and dimensions (i.e the structure) of the indicator definitions. No indicator values or metadata are included in this file. Each system need to start with the TemplateDSD.xml file and construct the specific DSD.xml file corresponding to the indicator definitions and the structure of the data to exchange.
Customization of the Codelists
  • Managing custom codelists - CL_AGROUP is a codelist with content specific to WHO. To reference this custom codelist in the DSD, the following changes are needed:

    1. isFinal="false" means that the list is "volatile" and can be modified often. A value of true would mean the list must be considered read-only for the version specified, so changing the content would mean the version must change.
    2. agencyID="WHO"
    3. urn="urn:sdmx:org.sdmx.infomodel.codelist.CodeList=WHO:CL_AGROUP[1.0]"
    4. uri="./CUSTOM/WHO/codelists/CL_AGROUP.WHO.1.0.xml"
  • Managing Common codelist overriding - The sample has an overriden version of the common code list CL_GENDER (to include fetus in the list) as there are some cases when you may need to ignore the common code lists proposed and create your own custom version. You can't modify any COMMON code-lists so you must always create your own custom version if you wish to deviate from the COMMON code-list. Eventually, these custom changes may be integrated into a future version of the common SDMX-HD standard.
Customization of the Hierarchies
  • HCL_COMMON_HY_HIF - This hierarchy was removed when the DSD was created from the TemplateDSD as none of the common hierachies are needed for this sample.
  • HCL_STATIC_CODELIST_HIERARCHIES - This hierarchy was removed when the DSD was created from the TemplateDSD as none of the common hierachies are needed for this sample.
  • HCL_COMMON_HY_UNIT - The following hierachy is needed to map Units of Measure to the Unit Groups they belong to.
  • HCL_CONFIGURATION_HIERARCHIES - The following hierachies are needed for this sample:

    1. HY_GEOGRAPHIC_PLACE_NAME - The structural relationship between Geographical places.

    2. HY_INDICATOR - For percent and ratio indicators, this hierarchy contains the structural relationship between the indicators and the type of operands (numerator and denominator) the relation adheres to. An example is the ratio indicator (67).
    3. HY_INDICATOR_DISAGGREGATION - The structural relationship between indicators and their disaggregations. Note that all indicators have a root disaggregation based on the type of value (CL_VALUE_TYPE): whether the data exchanged contains indicator values or targets.

Customization of the Concepts
  • CS_COMMON - This concepts scheme contains the list of common concepts created as dimensions to key to the observation (Yet these are not disaggregations).
  • CS_ATTRIBUTE - This concepts scheme contains the list of common attribute and metadata attribute concepts
  • CS_DISAGGREGATION - This concepts scheme contains the list of common disaggregation created as dimensions to key the observation
  • CS_CUSTOM_WHO - This concepts scheme contains the list of custom concepts for WHO
  • Should you need to add any new concepts specific to your data exchange, you should create a new concepts sheme and use it in a similar fashion as CS_CUSTOM_WHO
Customization of the Attributes Attributes reference concepts and optionally codelists. The concept name becomes the attribute name that appears in the CDS.xml file. Each attribute can be created as conditional or mandatory and associated to any one of the following levels:
  • ObservationLevel - Indicator values are stored at this level, so attributes that describe the indicator values can be linked here.
  • SeriesLevel - Observations are grouped into series by indicator so attributes that describe indicator can be linked here.
  • DataSetLevel - Here the attributes describe the overall content of the dataset.
Attributes can either be associated to a codelist or not. When no codelist is specified the TextFormat element must be specified to ensure the type of the attribute is known. This is important as the generation of the Compact Data Set schema validation file (CDS.xsd) requires the type to be known. Here is an example of TextFormat element to describe an non-codelist attribute of type "count":
<structure:TextFormat textType="Count"/>
MSD File - This file contains structural information about the metadata.
Customization of the Codelists
  • Managing custom codelists - Creating code-list as external files (instead of inline in the DSD) means that can easily be re-used if also needed for metadata. Whilst not recommended, if you inline a code-list in a DSD file and then wish to use it the "External" keyword must be used in the MSD RepresentationScheme to denote this.
     
    <structure:RepresentationScheme representationSchemeType="External" representationScheme="CL_INDICATOR" representationSchemeAgency="SDMX-HD" />
  • Managing Common codelist overriding - In this sample there are no overrides for the common code lists, however they are managed in the same way as the DSD.
Customization of the Concepts
  • CS_ATTRIBUTE - This concept scheme has a special group of generic concepts that are often required to describe data or metadata. Note that these concepts have been prefixed by underscore "_" to denote them as generic concepts. Here is the list of generic concept:

    1. _CODE - Used for Indicators codes 65,67,75
    2. _DESCRIPTION - Used for the Indicator Descriptions. Note that the codelist code for an Indicator has a "Description" element. However it is the Indicator name that is placed in the code list and here in the metadata we place the actual description.
    3. _CREATED_ON - Not actually used in this sample.
    4. _CREATED_BY - Not actually used in this sample.
    5. _UPDATED_ON - Not actually used in this sample.
    6. _UPDATED_BY - Not actually used in this sample.
    7. _ORDER_BY - Used to specify an order for Indicators. In this sample we always want to specify Numerators first and then Denominators which is why the order by for these two items is set accordingly.
    8. _IS_ARCHIVED - Not actually used in this sample.
    9. _IS_READONLY - Not actually used in this sample.
    10. _VERSION - The Indicator Definition version e.g 1.0
    11. _VSTATUS - Not actually used in this sample.

  • CS_CUSTOM_WHO - This concepts scheme contains the list of custom concepts used to describe metadata elements specific to WHO.
  • Should you need to add any new concepts specific to your organization, you should create a new concept scheme and use it in a similar manner as CS_CUSTOM_WHO.
Managing Identifier Component The MSD has IdentifierComponent elements to target which codelist should be used for the metadata. These codelist can be either the ones specified in the DSD (External) or the ones specified in the MSD (Inline or as a external file). You can imagine the IdentifierComponent as a way to reference the tables in a database, with the codelist specified containing the keys to each row.
  • Once the list of IdentifierComponent has been created (Forming the FullTargetIdentifier) a ReportStructure if create to denote the metadata you wish to report.
    1. IC_INDICATOR - Reference to the INDICATOR concept and used in the GMS by metadata reported values
    2. RS_INDICATOR - Represents the metadata attributes that the GMS file wil report for IC_INDICATOR.
Managing Metadata Attributes The MSD has MetadataAttribute elements which identify the metadata fields your metadata reported values will be associated to. You can imagine MetadataAttribute as a way to reference the columns values in a database. Each MetadataAttribute references a concept and optionally a codelist and has a usageStatus attribute to indentify if a MetadataAttribute is "Mandatory" or "Conditional". e.g. in the sample we specify metadata linked to Indicators. One of the metadata attributes is the Indicator type which is specified as Mandatory which means you must at least have reported values for this metadata.
GMS File - This file contains all the actual metadata values for MetadataAttributes linked to the IdentifierComponents
GMS to MSD Relationship At the start of each GMS file you have the following MetaDataSet attributes:
  • metadataStructureURI - This has the uri to the MSD file allowing the GMS and the MSD to be associated.
  • datasetID - This is simply an ID for this MetaDataSet
Following these attributes you have the following child elements of the MetaDataSet:
  • MetadataStructureRef - This is a reference to the MetadataStructureDefinition ID in the MSD
  • MetadataStructureAgencyRef - This is a reference to the MetadataStructureDefinition AgencyID in the MSD
  • ReportRef - This is a reference to the ReportStructure ID in the MSD
Managing AttributeValueSet An AttributeValueSet element is created for each metadata attribute. This element has a TargetRef element which references the FullTargetIdentifier ID in the MSD as well as TargetValues element that are used to reference the IdentifierComponent (Table) and the codelist value (Row) that the metadata is for. For example here is the XML that identifies the "Indicator 67 - Neonatal mortality rate (per 1000 live births)" which has a CL_INDICATOR code value of 0.

<gen:TargetValues><gen:ComponentValue component="IC_INDICATOR" object="Codelist">0</gen:ComponentValue></gen:TargetValues>
The actual metadata is contained within a ReportedAttribute element which references the concepts this data pertains to (this maps to the concept used in the MetadataAttribute). Note that when the data type is a string it has been encapsulated by a CDATA tag.
